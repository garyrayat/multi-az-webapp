# CLAUDE.md тАФ multi-az-webapp

Terraform project that deploys a production-grade, resilient web application on AWS across two
Availability Zones. The defining feature is a `lab_running` boolean that toggles the entire
billable stack on and off without destroying the VPC skeleton, making cost management trivial
during development.

## Architecture

Three-tier, multi-AZ across `us-east-1a` and `us-east-1b` by default.

```
Internet
   тФВ
   тЦ╝
ALB (public subnets тАФ 10.0.1.0/24, 10.0.2.0/24)
   тФВ  Layer-7 HTTP/HTTPS, health check /health every 30s
   тЦ╝
EC2 ASG (private subnets тАФ 10.0.10.0/24, 10.0.11.0/24)
   тФВ  t3.micro, Amazon Linux 2023, nginx, 2 desired / 1 min / 4 max
   тФВ  No SSH тАФ access via SSM Session Manager only
   тЦ╝
RDS PostgreSQL 16.3 (database subnets тАФ 10.0.20.0/24, 10.0.21.0/24)
   тФВ  db.t3.micro, gp3 20 GB (auto-scales to 100 GB), encrypted
   тФВ  Password never in tfstate тАФ generated by random_password, stored in Secrets Manager
   тЦ╝
AWS Secrets Manager
   тФФтФАтФА {project}/{env}/db-credentials  (JSON: username, password, host, port, dbname)
```

Traffic flow for AWS API calls from EC2 (when `lab_running=true`):
- S3 тЖТ free S3 Gateway Endpoint (bypasses NAT entirely)
- SSM, CloudWatch Logs, Secrets Manager тЖТ Interface Endpoints (private DNS, no NAT traffic)

## The `lab_running` flag

`lab_running` is the master cost switch. Default is `false` in `terraform.tfvars`.

### What each value means

| Resource | `lab_running=false` | `lab_running=true` |
|---|---|---|
| VPC, subnets, IGW | Always deployed | Always deployed |
| Security groups | Always deployed | Always deployed |
| IAM role + profile | Always deployed | Always deployed |
| S3 gateway endpoint | Always deployed | Always deployed |
| Budget + SNS | Always deployed | Always deployed |
| CloudWatch log groups | Always deployed | Always deployed |
| NAT Gateways (2├Ч) | **Destroyed** | Created |
| Private route table default route | **No route** | Via NAT Gateway |
| ALB + target group + listener | **Destroyed** | Created |
| EC2 ASG desired/min/max | **0 / 0 / 0** | 2 / 1 / 4 |
| RDS instance | **Destroyed** | Created |
| Secrets Manager secret | **Destroyed** | Created |
| Interface VPC Endpoints (5├Ч) | **Destroyed** | Created |
| CloudWatch alarms (3├Ч) | **Destroyed** | Created |
| CloudWatch dashboard | **Destroyed** | Created |

### Estimated monthly cost

| State | Approximate cost |
|---|---|
| `lab_running=false` | ~$2тАУ5/month (VPC, log groups, budget alerts) |
| `lab_running=true` | ~$50тАУ80/month (NAT ├Ч2, ALB, EC2, RDS, endpoints) |

### How it's implemented

Each module receives `lab_running` as a variable and uses `count` or `for_each`:

```hcl
# ALB тАФ count pattern
resource "aws_lb" "main" {
  count = var.lab_running ? 1 : 0
  ...
}

# ASG тАФ capacity pattern
resource "aws_autoscaling_group" "main" {
  desired_capacity = var.lab_running ? 2 : 0
  min_size         = var.lab_running ? 1 : 0
  max_size         = var.lab_running ? 4 : 0
  ...
}

# VPC NAT Gateways тАФ count pattern
resource "aws_nat_gateway" "main" {
  count = var.lab_running ? length(var.availability_zones) : 0
  ...
}

# VPC private route table тАФ dynamic block pattern (keeps route table, removes route)
resource "aws_route_table" "private" {
  dynamic "route" {
    for_each = var.lab_running ? [1] : []
    content {
      cidr_block     = "0.0.0.0/0"
      nat_gateway_id = aws_nat_gateway.main[count.index].id
    }
  }
}

# VPC Endpoints тАФ for_each pattern
resource "aws_vpc_endpoint" "interface" {
  for_each = var.lab_running ? local.interface_endpoints : {}
  ...
}
```

Modules that produce `null` outputs when `lab_running=false` use `try()` to avoid errors:
```hcl
output "alb_dns_name" {
  value = try(aws_lb.main[0].dns_name, null)
}
```

The root `main.tf` guards against null ARNs being passed downstream:
```hcl
# coalesce() converts null тЖТ "" so ASG target_group_arns list is never [null]
target_group_arn = var.lab_running ? coalesce(module.alb.target_group_arn, "") : ""
```

## Module details

### `modules/vpc`

Creates the complete network foundation. Always deployed.

Key resources:
- `aws_vpc` тАФ 10.0.0.0/16, DNS hostnames and support enabled
- `aws_internet_gateway` тАФ attached to VPC
- `aws_subnet.public[count]` тАФ one per AZ, `map_public_ip_on_launch=true`, tagged `Tier=Public`
- `aws_subnet.private[count]` тАФ one per AZ, tagged `Tier=Private`
- `aws_subnet.database[count]` тАФ one per AZ, tagged `Tier=Database`
- `aws_eip.nat[count]` тАФ only when `lab_running=true`
- `aws_nat_gateway.main[count]` тАФ one per AZ, only when `lab_running=true`
- `aws_route_table.public` тАФ default route via IGW
- `aws_route_table.private[count]` тАФ default route via NAT (only when `lab_running=true`)
- `aws_route_table.database` тАФ no default route (DB tier has no outbound internet)
- `aws_db_subnet_group.main` тАФ required by RDS, spans all database subnets

Outputs consumed by other modules: `vpc_id`, `public_subnet_ids`, `private_subnet_ids`,
`database_subnet_ids`, `db_subnet_group_name`, `private_route_table_ids`.

### `modules/security_groups`

Three-tier security groups. Always deployed. No `lab_running` gate тАФ SGs are free.

- `alb_sg` тАФ inbound 80/443 from 0.0.0.0/0
- `app_sg` тАФ inbound 80 only from `alb_sg` (SG reference, not CIDR)
- `db_sg` тАФ inbound 5432 only from `app_sg` (SG reference, not CIDR)

All three allow all outbound. No port 22 anywhere тАФ SSH access is intentionally absent.

### `modules/iam`

EC2 instance role and profile. Always deployed.

Three AWS managed policies attached:
- `AmazonSSMManagedInstanceCore` тАФ SSM Session Manager shell access
- `CloudWatchAgentServerPolicy` тАФ ship metrics and logs to CloudWatch
- `SecretsManagerReadWrite` тАФ read DB credentials at runtime

The instance profile (not the role itself) is referenced in the ASG launch template.

### `modules/alb`

Application Load Balancer. Gated: only when `lab_running=true`.

Resources (all use `count = var.lab_running ? 1 : 0`):
- `aws_lb` тАФ internet-facing, application type, spans all public subnets
- `aws_lb_target_group` тАФ port 80 HTTP, health check on `/health`, 2 healthy / 3 unhealthy thresholds, 30s interval
- `aws_lb_listener` тАФ port 80 тЖТ forward to target group

Outputs use `try()` for null safety: `alb_dns_name`, `alb_arn`, `alb_arn_suffix`,
`target_group_arn`, `target_group_arn_suffix`.

### `modules/asg`

Auto Scaling Group. Partially gated: launch template always exists; capacity scales to 0 when `lab_running=false`.

- `data.aws_ami.amazon_linux` тАФ latest `al2023-ami-*-x86_64` HVM AMI (dynamic lookup)
- `aws_launch_template` тАФ t3.micro, app SG, IAM profile, `user_data.sh` rendered via `templatefile()`
- `aws_autoscaling_group` тАФ private subnets, ELB health check type, 120s grace period

`user_data.sh` bootstraps nginx, creates an `/health` endpoint, and displays the AZ the instance
is running in on the index page. Used to visually verify multi-AZ routing.

ASG `desired_capacity = var.lab_running ? 2 : 0` тАФ instances terminate when lab is off.

### `modules/rds`

PostgreSQL RDS instance with Secrets Manager credentials. Fully gated: `count = var.lab_running ? 1 : 0`.

Password flow:
1. `random_password.db` тАФ 32 chars, special chars limited to `!#$%^&*()-_=+[]`
2. `aws_secretsmanager_secret` тАФ named `{project}/{env}/db-credentials`
3. `aws_secretsmanager_secret_version` тАФ stores JSON: `{username, password, host, port, dbname}`
4. EC2 fetches the secret at runtime via the IAM role тАФ password never in state or tfvars

RDS config:
- Engine: PostgreSQL 16.3
- Storage: gp3, 20 GB allocated, auto-scales to 100 GB
- `storage_encrypted = true` always
- `publicly_accessible = false` always
- `deletion_protection = false` and `skip_final_snapshot = true` for lab use
- `recovery_window_in_days = 0` on the secret (immediate delete, change to 30 for prod)
- `multi_az` defaults to `false` тАФ enable per environment for production

### `modules/vpc_endpoints`

Reduces NAT Gateway data transfer charges. Two types:

**S3 Gateway Endpoint** тАФ always deployed, free:
- Injects routes into all private route tables
- S3 traffic from EC2 bypasses NAT entirely

**Interface Endpoints** тАФ `for_each = var.lab_running ? local.interface_endpoints : {}`:
- `ssm`, `ec2messages`, `ssmmessages` тАФ SSM Session Manager connectivity
- `logs` тАФ CloudWatch Logs Agent
- `secretsmanager` тАФ Secrets Manager reads

All interface endpoints use `private_dns_enabled = true`, which transparently overrides DNS so
existing SDK calls route through the endpoint without code changes.

A dedicated security group allows inbound HTTPS (443) from the VPC CIDR to the endpoint ENIs.

### `modules/cloudwatch`

Observability layer. Log groups always deployed; alarms and dashboard gated.

**Always deployed:**
- `aws_sns_topic.alarms` тАФ operational alerts (separate from budget SNS topic)
- `aws_cloudwatch_log_group.app` тАФ `/{project}/{env}/app`, 7-day retention
- `aws_cloudwatch_log_group.nginx` тАФ `/{project}/{env}/nginx`, 7-day retention

**When `lab_running=true`:**
- Alarm: `asg-cpu-high` тАФ EC2 average CPU > 80% for 2├Ч 5-minute periods
- Alarm: `alb-unhealthy-hosts` тАФ any unhealthy target (threshold 0, fires immediately)
- Alarm: `rds-cpu-high` тАФ RDS CPU > 80% for 2├Ч 5-minute periods
- Dashboard тАФ 4 widgets: EC2 CPU, ALB request count, ALB healthy/unhealthy hosts, RDS CPU

All alarms publish to `aws_sns_topic.alarms` for both alarm and OK state transitions.

### `modules/budget`

Monthly cost guardrail. Always deployed (budget and SNS are free).

- `aws_sns_topic.budget_alerts` тАФ separate from operational alarms SNS topic
- SNS topic policy тАФ grants `budgets.amazonaws.com` permission to publish
- `aws_budgets_budget.monthly` тАФ COST type, MONTHLY, alerts at 50% and 80% of `budget_limit`
- Default: $100/month limit, alerts at $50 and $80

Email subscribers must click the AWS confirmation link before alerts are active.

## GitHub Actions workflows

### `terraform-plan.yml` тАФ triggered on every PR to `main`

Steps in order:
1. **Checkout** (`actions/checkout@v4`)
2. **Setup Terraform** тАФ pins to 1.14.0 (requires `use_lockfile` support)
3. **`terraform fmt -check -recursive`** тАФ fails if any `.tf` file is unformatted; `continue-on-error: true` (advisory)
4. **tfsec** (`aquasecurity/tfsec-action@v1.0.3`) тАФ static security analysis, SARIF output to GitHub Security tab; `soft_fail: true`
5. **`terraform init`** тАФ connects to S3 backend; requires `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` secrets
6. **`terraform validate`** тАФ syntax and type checking
7. **`terraform plan -var="lab_running=false" -var='alert_emails=[...]' -out=tfplan`** тАФ always plans with lab off for safety
8. **`terraform show -json tfplan > plan.json`** тАФ exports machine-readable plan
9. **`python3 scripts/cost_warning.py plan.json`** тАФ emits GHA `::warning::` annotations for expensive resources
10. **`python3 scripts/parser.py plan.json > plan_comment.html`** тАФ rich markdown table
11. **Delete previous plan comments** тАФ removes stale `<!-- terraform-plan-comment -->` comments from the PR
12. **Post plan comment** тАФ posts format/init/validate/plan status table + collapsible plan output

Required GitHub secrets: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`.

### `terraform-destroy.yml` тАФ manual `workflow_dispatch`

Safety gate: user must type `DESTROY` in the confirmation input or the job fails immediately.

Steps:
1. `validate-input` job checks the confirmation string
2. `terraform-destroy` job (needs `validate-input`):
   - Init тЖТ list current state тЖТ `terraform apply -var="lab_running=false" -auto-approve`
   - Prints remaining resources after apply
   - Posts destroy summary to GitHub Actions job summary

Using `terraform apply -var="lab_running=false"` (not `terraform destroy`) preserves the free-tier
skeleton (VPC, subnets, SGs, IAM, S3 endpoint, budget).

## Python scripts

### `scripts/cost_warning.py`

Reads `plan.json` (output of `terraform show -json`), iterates `resource_changes`, and emits
GitHub Actions annotations for creates/updates of:

| Resource type | Annotation level |
|---|---|
| `aws_nat_gateway` | WARNING |
| `aws_lb` | WARNING |
| `aws_db_instance` | WARNING |
| `aws_vpc_endpoint` | INFO |
| Large EC2 instance types (m5, c5, r5, t3.large+) in launch templates | WARNING |
| `aws_db_instance` with `multi_az=true` | WARNING |

Exits 0 always тАФ advisory only, does not block PR merges. Change `return 0` тЖТ `return 1` at the
bottom of `check_plan()` to make it blocking.

### `scripts/parser.py`

Reads `plan.json` and outputs a markdown PR comment with:
- Risk level badge (LOW/MEDIUM/HIGH based on destroy/replace counts)
- Summary table: add/change/destroy/replace counts
- Per-action sections, grouped by Terraform module, in collapsible `<details>` blocks
- Emoji icons per resource type (defined in `RESOURCE_ICONS` dict at top of file)

Output goes to both stdout (captured as `plan_comment.html` in CI) and `plan_summary.md`.

## Common operations

### Bring up the full stack

```bash
terraform apply -var="lab_running=true" -var='alert_emails=["you@example.com"]'
```

### Tear down billable resources (keep VPC skeleton)

```bash
terraform apply -var="lab_running=false"
```

Or trigger the GitHub Actions destroy workflow from the Actions tab.

### Fix format errors in CI

```bash
terraform fmt -recursive
```

### Connect to an EC2 instance (no SSH needed)

```bash
# Find instance ID in AWS Console тЖТ EC2 тЖТ Auto Scaling Groups тЖТ Instances
aws ssm start-session --target i-xxxxxxxxxxxxxxxxx
```

### Read DB credentials

```bash
aws secretsmanager get-secret-value \
  --secret-id multi-az-webapp/dev/db-credentials \
  --query SecretString --output text | jq .
```

### View ALB endpoint after apply

```bash
terraform output alb_dns_name
```
